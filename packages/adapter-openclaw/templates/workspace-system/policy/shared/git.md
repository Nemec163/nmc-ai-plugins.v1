# Shared Policy: Git iteration closeout (commit + push)

## Цель

Если агент изменил файлы в репозитории — эти изменения должны быть:

1) закоммичены локально
2) отправлены в удалённый репозиторий (`origin`)

...в конце той же итерации (т.е. перед тем как агент отправит финальный ответ пользователю).

## Определение «итерации»

Итерация = один агентский turn, в ходе которого были **созданы/изменены/перемещены** файлы под git.

Если turn был только разговорным (без изменений на диске) — commit/push не нужен.

## Обязательная процедура (строго)

Перед финальным ответом пользователю:

1. Проверить, есть ли изменения:
   - `git status --porcelain`
2. Быстро просмотреть, что именно меняется (минимум):
   - `git diff` (и при необходимости `git diff --staged`)
3. Стадия:
   - `git add -A`
4. Коммит:
   - `git commit -m "<scope>: <summary>"`
5. Пуш:
   - `git push origin <branch>`

Где `<branch>` зависит от текущего git flow (см. `./git-flow.md` и флаг `system/tasks/active/.kanban.json`).

## Формат сообщений коммита

- Рекомендовано: `<scope>: <summary>`
  - `policy: enforce commit+push iteration closeout`
  - `research: add OpenClaw 2026.2.17 report`
  - `docs: update NMC system architecture`

(Скоупы не обязательны, но помогают фильтровать историю.)

## Безопасность (нельзя коммитить)

- Секреты/токены/ключи/дампы auth.
- Любые файлы, которые попадают под `.gitignore` (особенно `state/`, `secrets/`, `system/config/secrets/`, `*/memory/**/*token*`, и т.д.).

Если есть подозрение на секреты:
- не пушить
- попросить подтверждение у человека (или вынести секреты в игнорируемый путь)

## Обработка ошибок

- Если `git push` не проходит (нет ключей/прав/сети):
  - всё равно сделать локальный commit
  - в ответе пользователю написать кратко: что закоммичено, но push не удался, и с какой ошибкой

## Опционально: автоматизация

Можно использовать скрипт:

- `system/scripts/git-iteration-closeout.sh "msg"`

Он делает add/commit/push при наличии изменений.
